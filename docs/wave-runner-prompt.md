# Wave-runner prompt

Той самий текст вставляється в КОЖНУ нову сесію (або крутиться лупом) — він сам знаходить,
що робити далі. Одна сесія = одна хвиля = свіжий контекст.

---

Прочитай CLAUDE.md і docs/smart-chat-instagram-plan.md.

Знайди ПЕРШУ хвилю з невиконаними чекбоксами за таким порядком пріоритету:
H1–H5 → A → B → C → D → G → E → H6–H11.

Виконай ТІЛЬКИ цю одну хвилю. Правила:

1. Працюй у гілці wave-<ідентифікатор хвилі> (створи від master, якщо нема).
   Перед стартом перевір git status: якщо в robочому дереві чужі незакомічені
   зміни у components/templates/** або lib/blocks/** — не чіпай ці файли,
   обходь або зупинись і скажи чому.
2. Після КОЖНОГО пункту: npx tsc --noEmit; познач чекбокс [x] у план-файлі;
   окремий коміт (conventional commits, англійською).
3. Дотримуйся HARD INVARIANTS з CLAUDE.md безумовно.
4. Наприкінці хвилі:
   - npm run build (має бути зелений);
   - живі Playwright-скріншоти зачеплених екранів (dev-сервер, lvh.me-хости;
     тест-юзер у CLAUDE.md), для AI-флоу — один реальний smoke-виклик;
   - adversarial codex review робочого дерева; підтверджені знахідки виправ
     і закоміть;
   - у план-файлі допиши коротку нотатку, якщо десь відхилився від плану.
5. ПОЛІТИКА ВІДХИЛЕНЬ. Ти маєш повну відповідальність за якість продукту в межах
   узгодженого напрямку (заявки в Telegram, радикальна простота для нетехнічного
   власника, розумний чат, українська мова). План — карта, не кайдани:
   - ТАКТИЧНІ відхилення роби сам, без питань: кращий спосіб реалізації, дрібні
     UX-покращення по дорозі, виправлення знайдених багів, до ~пів дня додаткової
     роботи. Кожне відхилення — нотаткою в план-файл (що і чому).
   - КАРДИНАЛЬНІ — зупинись і постав питання: зміна схеми даних чи флоу онбордингу,
     нові платні сервіси/залежності, видалення функціональності, все, що суперечить
     HARD INVARIANTS або помітно міняє продуктову поведінку.
6. Після ОДНІЄЇ хвилі — ЗУПИНИСЬ. Не починай наступну. Фінальним повідомленням
   дай: що зроблено, комміти, відхилення від плану, що відкладено, чи готова
   гілка до merge.

Якщо хвиля вже почата попередньою сесією (частина чекбоксів [x]) — продовж
з першого невиконаного пункту в тій самій гілці.

---

## Запуск

Разово (свіжа сесія на хвилю, найкращий контроль):
вставити промпт → дочекатись → переглянути → merge → нова сесія.

Автономний луп (кожна ітерація = свіжий процес і чистий контекст):

```bash
for i in 1 2 3 4 5 6 7 8; do
  claude -p "$(sed -n '/^Прочитай/,/до merge\./p' docs/wave-runner-prompt.md)" \
    --permission-mode acceptEdits
done
```

Для найважчих хвиль (E-Instagram, H6 QA-конвеєр) — додати в кінець промпту
слово `ultracode`: розкладе роботу на панель агентів з adversarial-перевіркою.

## Чому це ефективно

- Пам'ять між сесіями = план-файл (чекбокси) + коміти + CLAUDE.md, не контекст.
- Свіжий контекст на хвилю > одна мега-сесія з авто-компактом.
- Ідемпотентність: промпт можна запускати повторно — він продовжить з місця зупинки.
- Review gate (/codex:setup) вже увімкнений у репо — кожна сесія отримає
  автоматичне стоп-рев'ю понад пункт 4.
