# Вітрина — MVP Build-план (v1.1)

> **Мета:** живий згенерований сайт квіткарки на піддомені `назва.vitryna.com.ua`, зібраний через розмову з AI-агентом, який власник може відредагувати дотиком і опублікувати — **і з якого заявка падає власнику в Telegram**.
> **Критерій успіху:** нетехнічна людина за ~15 хв діалогу отримує опублікований пристойний сайт без сторонньої допомоги. Timebox ≈ 4–6 тижнів.
> **Скоуп:** архітектурний бриф §15 (MVP-скоуп v1). Рішення й «чому» — у брифі; тут — **порядок робіт**.
> **Стек:** Next.js (App Router, TS) на Vercel · Supabase (Postgres) · піддомени `*.vitryna.com.ua`.
> **Режим:** ship-fast. Кожна фаза закінчується видимою «перемогою», яку можна показати.

## Легенда
- 🏁 **Перемога фази** — те, що видно на екрані наприкінці.
- ⛓️ **Залежність** — що має бути готове раніше.
- 📄 — посилання на розділ брифу.

---

## Фаза 0 — Каркас (≈0.5 дня)
Ціль: порожній застосунок живе на Vercel, DNS налаштований.

1. `create-next-app` (App Router, TypeScript, Tailwind). Репо → Vercel-проєкт.
2. Купити/під'єднати `vitryna.com.ua`; DNS-запис **wildcard** `*.vitryna.com.ua` → Vercel (раз, не per-tenant). 📄 §6 Задача 1.
3. Supabase-проєкт; підключення (env). Zod у залежностях.
4. `middleware.ts` — кістяк розділення: платформні хости (`app.`, root, `www.`) vs tenant (решта). Поки просто лог/rewrite. 📄 §2.5.

🏁 `test.vitryna.com.ua` віддає заглушку «tenant test»; `app.vitryna.com.ua` — заглушку дашборду.

---

## Фаза 1 — Модель даних + мультитенант-рендер (≈2–3 дні)
Ціль: захардкоджений сайт у БД рендериться на піддомені. Це доводить хребет.

⛓️ Фаза 0.

1. **Міграції Supabase:** `tenants`, `pages`, `blocks`, `tenant_members`, `conversations`. 📄 §5.1.1, §3.1, §4.9.
   - `pages`: draft/published-версії контенту. 📄 §5.5.
2. **Реєстр блоків** (6 типів для MVP: `hero`, `services`, `gallery`, `testimonials`, `contacts`, `lead_form`) — компонент + Zod-схема кожного; `contacts` включає контакт-кнопки `tel:`/`t.me` (лінки); `lead_form` — єдиний блок зі серверним хендлером (📄 §5.6), у Фазі 1 хендлер = запис у БД (пуш — Фаза 4). 📄 §4.1, §15.
3. **Тема-токени** (`theme: {colors, fonts, ...}`) окремо від контенту; блоки читають CSS-перемінні. 📄 §4.5.
4. **Роут** `app/_sites/[host]/[[...slug]]/page.tsx` + rewrite у middleware (Host → canonicalHostname → `/_sites/{host}/...`). `await params`, optional catch-all, `notFound()` для невідомого slug. 📄 §2.2, §5.2, §2.4.
5. **PageRenderer**: `PageSchema.parse()` → мапа `blockType → компонент` → прохід масивом; `UnknownBlock` fallback. 📄 §4.1.
6. **ISR + тег-інвалідація** (`unstable_cache` + `revalidateTag('tenant:{host}')`), purge на «Опублікувати». 📄 §9.1. ⚠️ зафіксувати версію Next перед кодом.
7. Захардкодити 1 tenant + 1 page + блоки прямо в БД.

🏁 Реальний згенерований-руками сайт квіткарки видно на `kvity.vitryna.com.ua`, віддається з кешу.

## Фаза 2 — Field-схема вертикалі + генерація (≈2–3 дні)
Ціль: з набору фактів AI збирає валідні блоки. Петля генерації жива (поки без чату).

⛓️ Фаза 1 (реєстр блоків).

1. **Field-схема «квіткарня»** — обов'язкові/опційні поля, анотація кожного `fact | creative`. Це і критерій «достатньо інфи» для агента (Фаза 3). 📄 §4.4, §15 (критичний шлях).
2. **Пресет `florist`**: `[hero, services, gallery, testimonials, contacts, lead_form]` (воронка в кожному пресеті — інваріант §4.2).
3. **Генератор**: tool use / structured output → **валідований JSON блоків**; `type` як `enum`, `additionalProperties:false`; ретрай на помилку схеми. 📄 §4.3. ⚠️ звірити актуальний синтаксис structured outputs Anthropic.
4. **Grounding**: слоти заповнюються лише з фактів; `fact`-поля копіюються 1:1 + нормалізація (телефон/адреса), пост-звірка рядків. 📄 §4.4.
5. Прогнати захардкоджений об'єкт фактів → згенерувати → зберегти в БД → побачити рендер (Фаза 1).

🏁 Міняєш JSON фактів → отримуєш інший коректний сайт. Генерація детермінована й безпечна.

---

## Фаза 3 — Агент-чат онбординг (≈тиждень+, найбільша фаза)
Ціль: нетехнічна людина в розмові доводить сайт до генерації. Це фундамент MVP.

⛓️ Фаза 2 (схема — джерело істини й критерій завершення).

1. **Auth** (Supabase email+пароль) + прив'язка `tenant_members`; створення tenant. 📄 §3.1.
2. **Персистентність** розмови `conversations(tenant_id, site_id, messages[])` з 1-го дня. 📄 §4.9.
3. **Slot-filling агент:**
   - питає **батчами** (не 15 окремих питань); заповнює об'єкт фактів за схемою; last-wins при корекції. 📄 §4.9.
   - «досить» = обов'язкові поля схеми заповнені; **ескейп-хетч «просто згенеруй»** → керована деградація (заглушки/опустити блоки). 📄 §4.9, §4.8.
4. **Крок підтвердження = редагована префілена форма** (не стіна тексту). На «Підтвердити» → об'єкт фактів іде в генератор (Фаза 2).
5. **Генерація піддомену** з української назви (транслітерація + перевірка колізій). 📄 §4.4.
6. **Еталонні діалоги** (10–15) як тест-набір — щоб ітерувати промпти не наосліп на друзях. 📄 §4.9.

🏁 Заходиш як нова людина → балакаєш з агентом → отримуєш свій сайт на піддомені.

---

## Фаза 4 — Воронка заявок + редагування + публікація (≈4–6 днів)
Ціль: власник правит контент, публікує — і отримує заявку в Telegram.

⛓️ Фаза 1 (схеми блоків), Фаза 3 (auth).

1. **Дотик-редагування**: натиск на елемент → **bottom-sheet панель** з полем зі схеми блоку + «Зберегти». Тільки цей елемент, без drag/resize. 📄 §3.
2. **Draft → «Опублікувати»** → purge тегів tenant. 📄 §5.5, §9.1.
3. **Завантаження фото**: input з галереї → обробка на клієнті (HEIC→, EXIF-orientation, стрип GPS, стиснення) → Storage → resize/WebP. 📄 §4.8.
4. **Воронка заявок** 📄 §5.6:
   - абстракція «канал повідомлень» (інтерфейс → Telegram / email; Viber/SMS — пізніші реалізації);
   - один центральний **Telegram-бот**: кнопка «Підключити Telegram» у дашборді → Start → зберігаємо `chat_id` у tenant config;
   - сабміт `lead_form` → запис у БД → пуш у Telegram власника + **email-fallback** (якщо Telegram не підключено);
   - заявки видно в дашборді (простий список).
5. (Опційно для тесту) «Перемкнути шаблон» — якщо лишиться час. 📄 §4.7.

🏁 Повна петля цінності: розмова → сайт → правки дотиком → фото → публікація → **тестова заявка прилетіла в Telegram**.

## Фаза 5 — Дати друзям
Ціль: перший живий фідбек.

1. 3–5 реальних людей роблять свій сайт.
2. **Пряме спостереження** (не аналітика — §12): де завмирають у чаті? витягує агент достатньо фактів? кажуть «так, це я»? чи зрозуміло їм підключення Telegram?
3. Північна метрика: чи доходять до опублікованого сайту із заявкою-в-Telegram, за скільки хвилин (ціль ~15 хв діалогу).

🏁 Знаєш, чи працює на живих нетехнічних людях і генерація, і воронка.

## Фаза 2 продукту (після MVP) — retention-драбина
> Деталі й оцінки — у `feature-ideas.md`; межі — §15.

1. **Мінімальна аналітика** (перегляди, сабміти, кліки-на-телефон) — передумова звіту.
2. **Щомісячний звіт простою мовою** (+ Google Search Console API для запитів).
3. **AI-відповідач на заявки** — лише з opt-in власника + guardrails (без цін/гарантій/строків, підпис «автовідповідь»).
4. **«Оновлюй мене сам»**-бот (той самий Telegram-канал) + фото-через-бота (📄 §4.8 v1.1).
5. **Запит-на-запис** (форма з бажаним часом, не календар) — ніші послуг.
6. **GBP-concierge** — обмежена одноразова послуга (не модель делівері).
7. Далі: білінг (UAH), Viber-пуш, власні домени/реєстрація, GBP API.
