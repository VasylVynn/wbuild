# План повного редизайну UI/UX + агентний редактор

> **Дата:** 2026-07-14. **Статус:** на затвердження власником.
> **Процес:** 2 паралельні research-агенти (UI-аудит усіх платформенних екранів + мапа архітектури редактора/AI) → драфт плану → адверсарний агент-валідатор по коду. Всі must-fix валідатора інтегровані.
> **Мандат власника:** аудиторія 30+ (не лише 50+); має виглядати класно і на мобільному, І на десктопі; без «величезних кнопок і емодзі»; чат-бот — розумніший і глибший (латентність прийнятна); редактор — «повноцінний», з device-preview і агентним чатом (інженер + SEO + маркетолог + бізнес-аналітик в одному), який пам'ятає всю історію.
> **Конфлікт-guard:** паралельний агент працює над шаблонами/секціями — цей план НЕ торкається `lib/templates/**`, `lib/blocks/skins.ts`, `components/site/**` (візуально), шрифтів tenant-сайтів. `PageRenderer` використовуємо read-only.

Архітектурні факти, що визначають рішення:

- **Preview редактора — inline React, не iframe** (`EditorShell.tsx:297-404`). Звуження контейнера НЕ перемкне Tailwind-брейкпоінти (`sm:`/`md:` дивляться на вікно браузера) → справжній device-preview можливий **лише через iframe** з власним viewport.
- **Розмова онбордингу сиротіє після finalize**: `generateAndPublish` створює НОВИЙ tenant по host, а `conversations.tenant_id` лишається на placeholder-tenant (`host=null`). Транскрипт живий у БД, але з редактора недосяжний — нема ключа. Це головний фікс для пам'яті агента.
- Per-block AI-поле (`aiEditBlockAction`) — stateless, один блок, один інпут, результат у форму. Для «агента-співрозмовника» непридатне як фундамент UX, але його механіка (форсований tool з per-block Zod-схемою) переюзається як внутрішній інструмент.
- `saveDraftBlocks` + `validateBlocks` вже вміють прийняти **довільний валідний масив блоків** — add/remove/reorder/edit виражаються одним записом. Готовий «двигун» для інструментів агента.
- Extended thinking сьогодні лише в генерації (budget 6000); чат — request/response без стрімінгу (індикатор «друкує…» — фейковий).

## 2. Дизайн-напрям (спільний для всіх фаз)

- **Тон:** професійний інструмент, не «додаток для бабусі»: щільність як у сучасних SaaS (Linear/Notion-клас), але зберігаємо теплу палітру бренду (ink/brand/honey) і радіуси — токени кольорів НЕ міняємо.
- **Іконки замість емодзі:** `lucide-react` (нова залежність, tree-shakeable). Емодзі лишаються тільки в контенті, де вони семантичні (напр. мок Telegram-повідомлення на лендінгу).
- **Розміри:** дефолт контролів ~40–44px (не 56), тексти 14–16px у UI, 48px-таргети лишаються на мобільному через responsive-класи, а не як глобальний дефолт.
- **Обидва viewport'и first-class:** кожен екран проєктується у двох станах — mobile (одна колонка, sheet-патерни) і desktop (сітки, таблиці, панелі, hover/focus-стани, клавіатура).

## 3. Фази

### P0 — Дизайн-фундамент (~1 день)

1. **Фліп дефолту `Button` `lg`→`md`** (`sm`/`md` вже існують — додавати нічого не треба). ⚠️ Свідомий побічний ефект: мовчки перестилює дефолтні кнопки в ~18 файлах (логін, онбординг тощо) — це і є мета, але пройтись оком по всіх.
2. Щільність — саме **хардкоджені px**, не `:root`: `btnSizes` (min-h-14/text-[18px]), `fieldBase text-[17px]`, `EmptyState py-14`, розміри чипів. Base font 17→16px — додатково, але сам по собі майже нічого не міняє.
3. `lucide-react` + заміна емодзі-іконографіки: тулбар редактора (🎨🖼️↻✏️), контроли секцій (👁🙈↑↓), адмін-навігація (🎨🧩🧪), лендінг-фічі.
4. `EmptyState`: `emoji` → `icon` (3 call sites: leads, sites, admin/generate).

### P1 — App shell + базові екрани (~3 дні)

1. **Постійна навігація:** десктоп — лівий sidebar (лого, Панель / Сайти / Заявки / Адмінка, юзер-меню знизу); мобільний — верхній бар + меню. Прибрати hub-and-spoke «← Панель».
2. **Справжній дашборд:** метрики з уже наявних `site_events` + `leads` (перегляди/заявки за 7/30 днів), картки сайтів зі статусом, швидкі дії. Не сплеш.
3. **Логін:** split-panel на десктопі (бренд-панель + форма), компактний тумблер. **Нове: скидання пароля** — `resetPasswordForEmail` + сторінка `/reset` + конфіг email-redirect (це справді новий флоу, його нема).
4. **Заявки:** десктоп — таблиця (колонки: контакт, сайт, повідомлення, дата) + пошук/фільтр; телефон — звичайне посилання, не pill-CTA. Мобільний — картки як зараз, компактніші.
5. **Сайти:** responsive-сітка карток, компактні секондарі, hover-стани.

### P2 — Каркас редактора: inspector + device-preview (~3–4 дні)

1. **≥lg: права inspector-панель** замість bottom-sheet (форма блоку, скіни, AI-хендл). Мобільний лишає sheet. `BlockSheet` розпадається на «контент форми» (спільний) + два контейнери.
2. **Device-preview: iframe.** Новий маршрут `/edit/[host]/frame`, який рендерить DRAFT-блоки через `PageRenderer` standalone. Перемикач Desktop / Tablet (768) / Mobile (375) задає ширину iframe — справжні брейкпоінти, `position:fixed` nav шаблонів працює коректно відносно iframe-viewport.
   - ⚠️ **Must-fix валідатора:** маршрут НЕ може жити під `(protected)/layout.tsx` — інакше email/Вийти-хром відрендериться **всередині** iframe. Окрема route group із мінімальним layout, але з тим самим `getUser`-гейтом. Same-origin cookies покривають auth.
   - ⚠️ **Синхронізація:** не reload-on-save (втратимо миттєвий optimistic preview), а `postMessage(blocks)` з редактора у frame після кожної локальної зміни.
   - **Вибір блоку** кліком у preview через iframe = зворотний `postMessage` (highlight + select). Обсяг залежить від відповіді на відкрите питання №2 (нижче).
3. Тулбар редактора: іконки+лейбли, Publish — праворуч, без емодзі.

### P3 — Агентний чат редактора (~6–8 днів, найбільша фаза)

**UX:** ліва колапсована чат-панель на десктопі (preview по центру, inspector праворуч); на мобільному — окрема вкладка/повноекранний sheet. Один агент, що поєднує ролі: інженер (структура/блоки), SEO-порадник, маркетолог (копірайт/CTA), бізнес-аналітик (що додати для конверсії). Розмовний, задає уточнення, пояснює що і чому зробив.

**Пам'ять (must-fix №1 — без цього фіча мертва):**
- Протягнути `conversationId` через `finalizeAction` (зараз сигнатура його НЕ приймає) → після `generateAndPublish` зробити `UPDATE conversations SET tenant_id = <реальний tenant.id>`. Placeholder-tenant лишається сиротою — нешкідливо.
- Нова міграція `0006_editor_chats.sql`: `editor_chats(id, tenant_id FK, messages jsonb, created_at)`.
- Контекст агента: `tenants.facts` + онбординг-транскрипт (тримований) + історія editor_chat + поточні блоки + статистика `site_events`.

**Інструменти агента (95% — обгортки наявних actions):**
- read: `get_site_state` (блоки+тема+факти+статистика);
- write у DRAFT через шлях `validateBlocks`: `update_block`, `remove_block`, `reorder_blocks`, `toggle_block`, `set_skin`, `switch_pack`, `switch_theme`;
- **нове:** `add_block` і `regenerate_block` — одноблоковий генератор (зараз `generate.ts` компонує лише цілу сторінку; це найбільший новий AI-код фази);
- поради (SEO/маркетинг) — без мутацій.

**Безпека (must-fix №2 — робити ДО інструментів):** інваріант §4.8 «жодних чужих image-URL» сьогодні живе ЛИШЕ в генерації (`groundImages`); `saveDraftBlocks` приймає будь-який URL (`assetUrl = z.string()`). Агент пише в draft без людської форми-гейта → **додати стрип не-bucket URL у draft-save шлях** (заодно закриє і ручний вектор). Зміни агента застосовуються до draft миттєво (як ручні), кожна — рядком у чаті («✓ Прибрав секцію Команда»), undo через наявний pocket. Publish — тільки людина.

**Модельний контур (must-fix №3):** extended thinking **несумісний** із форсованим `tool_choice` (задокументовано в `generate.ts:199-201`). Тому: основний цикл агента — thinking ~3000 + `tool_choice: auto`; точкова валідація props блоку — окремий суб-виклик без thinking із форсованим per-block Zod-tool (переюз механіки `edit-block.ts`).

**Транспорт:** SSE route handler (server action стрімити не вміє) + явний `maxDuration`; новий rate limit `editor_chat` ~120/год у `LimitName`+`DEFAULTS`.

### P4 — Онбординг-чат: розумніший і двопанельний (~3 дні)

1. **Справжній стрімінг** відповіді (route handler; фейковий «друкує…» на смітник). ⚠️ Guard «хід без питання» зараз робить corrective-retry ПІСЛЯ повної відповіді — зі стрімом несумісний: буферити до перевірки або перенести guard у промпт.
2. **Extended thinking ~2000 на хід** — глибші, доречніші питання (власник свідомо приймає латентність заради якості даних).
3. **Багатша стратегія порадника:** УТП, конкуренти, соцмережі, поради щодо фото — більше даних для генерації і для майбутнього агента редактора.
4. **Десктоп — дві панелі:** чат ліворуч, праворуч — «сайт народжується»: скелет-preview, що заповнюється фактами в реальному часі (назва → шапка, послуги → картки). Мобільний — одна колонка як зараз.

### P5 — Поліровка щільності (~1–2 дні)

Адмін-таблиця вміщається ≤1280px без h-скролу; прохід hover/focus/keyboard по всіх екранах; порожні стани; консистентність відступів.

## 3.1 Стек інструментів (безкоштовні, MIT — рішення 2026-07-14)

Повний UI-фреймворк (Mantine/HeroUI/Chakra) — **ні**: конфліктує з власною токен-системою. Натомість точкові шари:

| Шар | Що дає | Де в плані |
|---|---|---|
| **shadcn/ui** (Radix + Tailwind, copy-paste у репо, стилі на наших токенах) | Dialog/Popover/Tabs/Tooltip/Table + a11y (focus-trap, клавіатура) безкоштовно | P1 (таблиці, меню), P2 (inspector) |
| **Vercel AI SDK** (`ai` + `@ai-sdk/anthropic`) | `useChat`/`streamText`: SSE-стрімінг, стан, streaming tool calls, thinking-рендер — замість ручного SSE | P3 + P4 (обидва чати) |
| **AI Elements / assistant-ui** | Готовий скелет інтерактивного чату (markdown, tool-виконання видиме) | P3 + P4 |
| **react-resizable-panels** | Трипанельний редактор із перетягуваними роздільниками | P3 |
| **motion** (framer-motion) | Мікроанімації: поява повідомлень, stagger, transitions | P0+, скрізь потроху |
| **lucide-react** | Іконографіка замість емодзі | P0 |

**Межа:** AI SDK — лише для чатів. Генерація сайту (`generate.ts`, форсований tool_choice + grounding) лишається на прямому Anthropic SDK.

## 4. Відкриті питання власнику (потрібне рішення до старту відповідної фази)

1. **SEO-роль агента: порадник чи актуатор?** У схемах блоків нема SEO-полів (meta/title/OG per-block). Або SEO-роль обмежується реальними важелями (копірайт, структура, порядок секцій, alt-тексти), або додаємо SEO-поля в схему — це вже зміна словника блоків (зона template-агента → координувати). *(до P3)*
2. **Tablet/mobile preview — read-only чи editable?** Read-only = дешевше (без postMessage-selection у frame); editable = повний UX. *(до P2)*
3. **Черга конфлікт-зони:** шрифти tenant-сайтів (R1) і десктоп-поліровка залишаються за template-агентом чи повертаються сюди після його завершення? *(координація)*

## 5. Підсумкова оцінка

| Фаза | Обсяг | Оцінка |
|---|---|---|
| P0 | Дизайн-фундамент | <1 дня |
| P1 | Shell + дашборд + логін/reset + списки | 3 дні |
| P2 | Inspector + iframe device-preview | 3–4 дні |
| P3 | Агентний чат редактора + пам'ять + безпека | 6–8 днів |
| P4 | Онбординг: стрімінг + thinking + дві панелі | 3 дні |
| P5 | Поліровка | 1–2 дні |
| **Разом** | | **~3 тижні** |

Порядок жорсткий: P0→P1→P2→P3 (P3 залежить від inspector/чат-панелі P2), P4 можна паралельно з P3 (інші файли), P5 — фінал.

**Ризики:** (а) одноблоковий генератор — новий AI-код із власними едж-кейсами; (б) postMessage-синхронізація preview — акуратна, але добре відома механіка; (в) P0-фліп дефолту кнопки зачіпає всі екрани разом — після нього швидкий візуальний прохід обов'язковий; (г) стрімінг ламає fake-typing UX онбордингу — переробляється разом (P4).
