## 1. Контекст продукту

**Що це:** AI-конструктор сайтів. Користувач вводить деталі про свій бізнес → система генерує готовий сайт із доменом.

**Аудиторія:** українські власники малого бізнесу, нетехнічні, орієнтовно 50+ (квіткарні, барбершопи, юристи, локальні послуги тощо).

**Ключова теза проєкту:** генерація сайту — це ~20% продукту і ~100% демо. Вона виглядає як суть, але насправді найлегша частина. Уся складність — в інфраструктурі навколо: редагування, домени, мультитенантний хостинг, білінг у гривні, підтримка, економіка.

## 2. ✅ Архітектурний хребет (наскрізний принцип)

Чотири рішення, від яких залежить усе інше. Вони мають дотримуватись скрізь.

### 2.1. Контент = дані, а не HTML
Сайт користувача зберігається як **структуровані дані** (поля, списки, налаштування), а не як згенерований HTML/CSS.

**Наслідки, які тримати в голові:**
- Дозволяє редагувати контент через прості поля, не чіпаючи верстку.
- Дозволяє автоматично будувати SEO-розмітку (structured data) з тих самих полів.
- Дозволяє перегенерувати дизайн / поміняти шаблон, не торкаючись контенту. (Технічно це забезпечується тим, що **дизайн зберігається окремо від контенту як design tokens** — див. 4.5.)
- Дозволяє годувати ті самі дані в інші канали (сповіщення тощо) пізніше.

### 2.2. Один застосунок рендерить усі сайти (мультитенантність)
Ми **НЕ** хостимо N окремих сайтів. Ми хостимо **один** застосунок, який рендерить усі сайти з даних.

- «Сайт клієнта» — це **НЕ папка з кодом і НЕ окремий деплой. Це рядок у базі.** 5000 клієнтів = 5000 рядків у Supabase, а не 5000 застосунків. Код однаковий, відрізняються лише **дані**. Аналогія: користувачі в застосунку (один білд, дані юзера визначають, що показати) — тільки тут «хто це» визначається не логіном, а **доменом**.
- Запит → читаємо `Host`-заголовок → визначаємо tenant → тягнемо контент цього клієнта з БД → рендеримо. Той самий код обслуговує всіх.
- Структура роутів у Next: **немає файлів-під-клієнта**. Один catch-all роут (`app/_sites/[host]/[[...slug]]/page.tsx`) + `middleware.ts` (Host → rewrite у внутрішній шлях `/_sites/{host}/...`) + `lib/getTenant` + `lib/getPage`. Додати 5000-го клієнта → **0 нових файлів**.
- Деплоїмо **раз**. Фікс бага = push один раз → усі сайти оновлюються миттєво (це фізично один застосунок).

### 2.3. ✅ Data-driven рендер, а НЕ pre-generated файли
Є дві принципово різні моделі. Зафіксовано **Модель A**. Агент не повинен дрейфувати в бік B.

- **Модель A (обрано):** один data-driven застосунок рендерить кожен сайт із БД на льоту і **кешує** результат. Сайт = дані.
- **Модель B (відкинуто):** генерувати реальні статичні HTML-файли під кожного клієнта і складати на CDN. Сайт = набір файлів.
- **Чому A:** глобальний фікс/редизайн у B = перегенерувати всі сайти; складніша динаміка, інвалідація й сховище. ISR/кеш у моделі A дає швидкість статики (плюс B) **без** його мінусів — рендер один раз, віддача з кешу як статика, джерело — один застосунок + дані.

### 2.4. ✅ Tenant визначається на запиті через канонічний hostname `🔶→✅ (2026-06-30)`
Наскрізний інваріант, доданий за підсумком валідації роутингу й SEO. Він одночасно тримає правильний роутинг, коректне per-tenant SEO без втрати кешу і чисту прив'язку нав/секцій до даних.

- **ІНВАРІАНТ:** у моделі tenant є поле **`canonicalHostname`** — один обраний host клієнта (піддомен або власний домен). Усі похідні URL (canonical, `og:url`, посилання в sitemap, `url`/`image` у JSON-LD, `metadataBase`) будуються **з `canonicalHostname`**, а **не** з host поточного запиту і **не** з глобального домену платформи.
- **ІНВАРІАНТ:** host у рендер-шляху береться з **rewrite-`params`** (сегмент `[host]` після middleware-rewrite), а **не** з `headers()`. `headers()` переводить сторінку в динамічний рендер і вбиває ISR/кеш. Виняток — `app/sitemap.ts` / `app/robots.ts`, де `params` недоступні: там легітимно `headers().get('host')` (це навмисно робить їх динамічними per-host; запити в БД кешувати).

> *Відкинуто:* брати host з `headers()` усюди задля простоти — ламає кешування на тисячах сайтів. Журнал #1.

## 3. ✅ Редагування контенту (модель UX)

**ВИРІШЕНО: редагування через ПОЛЯ (форма / CMS-стиль), а не канвас-редактор.**

Причина — не лише дешевизна розробки. Для цієї аудиторії форма **краще за канвас по UX**: полотно, яке можна «зламати», їх лякає; поле «Телефон: [___]» — ні.

**ІНВАРІАНТ (зв'язок зі схемою блоків):** оскільки кожен блок має схему слотів (див. 4.1), **форма редагування генерується зі схеми блоку автоматично** (підхід Puck: `fields` → UI-форма → серіалізація назад у `{type, props}`). Один реєстр блоків обслуговує і рендер, і валідацію, і опис для AI, і форму редагування — їх не можна розсинхронізувати.

**Мінімальний набір редагування, що покриває ~90% потреб без канваса:**
1. Повторювані елементи списку — додати/прибрати послугу, товар, відгук, члена команди.
2. Сховати/показати цілу секцію.
3. Переставити елементи всередині списку.
4. Замінити зображення.
5. ✅ **«Перемкнути шаблон» `🔶→✅ (2026-06-30)`** — обрати інший кураторський пресет; контент-факти переносяться у слоти нового каркасу по реєстру блоків (нерозпізнане лишається в чернетці). Це **клапан тиску** під ризик первинної генерації (нижче) — без нього невдалий перший вибір замикає нетехнічного користувача й гробить онбординг. Перенесено з post-MVP у MVP. Це **перемикання пресету, а не редагування макета** — канвасом не стає.

> *Відкинуто:* тримати «перемкнути шаблон» у post-MVP як «дорогу операцію». Валідація показала, що без нього структурну помилку генерації користувач не виправить полями — пряма причина відмов. Журнал #5.

Усе, що далі (переставляти **секції** вручну, міняти кольори/макет руками), вже тягне справжній редактор → поза скоупом MVP.

⚠️ **Плата за це рішення:** планка якості *первинної* генерації зростає. Якщо людина править лише значення полів, то структуру/макет/дизайн AI мусить вгадати з першого разу — структурну помилку користувач не виправить **полями**. Саме тому «перемкнути шаблон» (п.5) втягнуто в MVP як страховку.

🔶 **ВІДКРИТЕ:** field-схеми під кожен тип бізнесу (вертикаль) ще не спроєктовані. Це і є прихована вартість продукту — не редактор, а бібліотека шаблонів + їхніх контент-моделей. Універсальна схема буде або занадто жорсткою, або занадто прісною → потрібні окремі схеми на вертикаль.

## 4. Генерація сайту

✅ **Підхід:** система блоків — AI обирає компоновку з **кураторських шаблонів** і заповнює слоти контентом + підбирає палітру (модель Durable / Wix ADI / Framer AI). **НЕ** вільна генерація сирого HTML/CSS (нестабільно, криво).

### 4.1. ✅ Реєстр блоків — єдине джерело істини `🔶→✅ (2026-06-30)`
Технічна основа генерації, додана за підсумком валідації.

- **Блок (секція)** = пара: **React-компонент** + **схема його слотів/полів** (напр. Zod). Сторінка = впорядкований **масив інстансів блоків** з даними: `[{type:'hero', props:{...}}, {type:'services', props:{items:[...]}}, ...]` (патерн Strapi Dynamic Zones / Puck / Sanity Portable Text).
- **Реєстр** мапить `blockType → компонент` і `blockType → схему`. Рендер = прохід масивом + валідація `PageSchema.parse()` **до** рендеру (невідповідність ловиться як помилка, не падає в рантаймі). `UnknownBlock` як fallback, щоб сайт ніколи не падав.
- **ІНВАРІАНТ:** усю генерацію блоків валідувати проти реєстру. Блок або поле поза реєстром — **помилка, а не «творча вільність»**. Той самий реєстр обслуговує рендер + валідацію + опис для AI + форму редагування (3) + нав-якорі (5.3).

> *Відкинуто:* вільну генерацію розмітки без реєстру — давала неіснуючі блоки/поля (галюцинації структури) і робила `switchTemplate` недетермінованим. Журнал #6.

### 4.2. ✅ AI робить рівно два рішення (пресет + слоти) `🔶→✅ (2026-06-30)`
Для надійності першої генерації AI **не збирає сайт із нуля вільною композицією**, а:
1. **Класифікує вертикаль** → обирає **кураторський пресет-каркас** (набір + порядок блоків + тема під вертикаль; напр. `florist`: `[hero, services, gallery, testimonials, contacts]`). `type`/вертикаль — `enum` із замкнутого списку.
2. **Заповнює слоти** фактами користувача.

> *Відкинуто (на MVP):* вільну композицію (AI сам обирає й сортує блоки з нуля) — ширший простір помилки, нижча надійність першої генерації. Лишається на «розширений режим» пізніше. Журнал #6.

### 4.3. ✅ Структурований вивід, а не сирий текст `🔶→✅ (2026-06-30)`
- **ІНВАРІАНТ:** AI повертає **валідований структурований JSON**, що відповідає схемі сторінки, а не HTML. Надійний базовий механізм — **tool use з `input_schema`** (модель повертає аргументи інструмента `build_page`); `type` як `enum` + `additionalProperties:false` фізично не дають вигадати неіснуючий блок чи поле. Опис доступних блоків для AI генерувати з тих самих схем (`zod-to-json-schema`), щоб опис і валідатор не розходились.
- Запобіжник: після відповіді `PageSchema.parse()`; помилка → повернути моделі текст помилки на 1–2 ретраї.

⚠️ **Звірити перед інтеграцією (статус міг змінитися):** Anthropic **structured outputs** — сильніша надбудова (constrained decoding). На момент написання — beta; статус (GA/beta) і точний синтаксис (`output_config.format` vs старий `output_format` + бета-хедер) звіряти в актуальних docs Anthropic, **не кодити з пам'яті/туторіалів**. Базовий, гарантовано доступний шлях — tool use з `input_schema`.

### 4.4. ✅ Контроль галюцинацій КОНТЕНТУ (grounding) `🔶→✅ (2026-06-30)`
Розділяємо два різні рівні — і це критично для бізнесу:
- **Структура** (вибір/порядок блоків, палітра) — креатив AI, галюцинація прийнятна.
- **Факти** (послуги, ціни, телефон, адреса, відгуки) — **тільки з анкети користувача**. Єдине дозволене джерело фактів — структурована форма вводу.
- **ІНВАРІАНТ:** маркетинговий «обгортковий» текст (хедлайни, заклики) AI писати може; реквізити/ціни/контакти — копіюються 1:1 і **пост-валідуються прямим порівнянням рядків**, а не довірою моделі. Структуру обирати окремим кроком від мапінгу фактів.

> ⚠️ Structured output гарантує лише *форму*, не *правду*: модель може ідеально валідно вписати вигадану ціну. Тому grounding (4.4) обов'язковий і незалежний від валідації схеми (4.3).

### 4.5. ✅ Тема/палітра/типографіка як design tokens `🔶→✅ (2026-06-30)`
- **ІНВАРІАНТ:** дизайн зберігається **окремо від контенту** як набір токенів (`theme: {colors, fonts, radius, spacing}`). Блоки **не містять** кольорів/шрифтів — читають токени через CSS-перемінні / ThemeProvider. «Перегенерувати дизайн, не торкаючись контенту» (2.1) = замінити об'єкт `theme`, масив `blocks` недоторканий. AI для дизайну повертає **тільки** об'єкт `theme` за схемою з обмеженими значеннями.

### 4.6. ⚠️ Усвідомлені ризики генерації
- **Галюцинації структури** — закрито реєстром (4.1) + структурованим виводом (4.3).
- **Галюцинації контенту** — закрито grounding (4.4).
- **Зображення:** сток виглядає генерично; AI-генерація дорога й не відповідає реальному товару (квіткарка хоче *свої* букети); змусити аудиторію 50+ завантажити нормальні фото — окремий бій.
- **Версіонування схем блоків:** старі сайти збережені під старою схемою. Поле `schemaVersion` у блоці + міграції (поле спершу `optional`, потім міграція; старі компоненти тримати, поки не мігровано всі сайти).
- ⚠️ **scaled-content-abuse:** той самий пайплайн дозволяє масово штампувати тонкі/дублікатні сторінки — ризик SEO-спаму й репутації кореня. Детально й мітигації — у 10.4.

🔶 **ВІДКРИТЕ:** підхід до зображень не вирішено (сток vs AI-ген vs обовʼязкове завантаження vs гібрид).

## 5. ✅ Структура клієнтського сайту

«Сайт» — це не одна сторінка. Дані живуть на **двох рівнях**, і це дві різні сутності.

### 5.1. Дворівнева модель даних
- **Рівень сайту (tenant config):** бренд/лого, назва бізнесу, пункти навігації, футер (контакти, години, соцмережі, копірайт), вибраний шаблон + палітра (як design tokens, 4.5), **`canonicalHostname`** (2.4), `nav_mode`. Один запис на клієнта, **спільне для всіх сторінок**. Рендериться оболонкою (layout), що обгортає будь-яку сторінку tenant.
- **Рівень сторінки:** список блоків/секцій конкретної сторінки.

Хедер/футер/нав НЕ копіюються в кожну сторінку — вони з рівня сайту.

### 5.1.1. ✅ «Сторінка» — завжди сутність, навіть для one-page `🔶→✅ (2026-06-30)`
- **ІНВАРІАНТ моделі даних:** у схемі **завжди** є сутність `pages`, навіть для односторінкового MVP — там просто **одна** сторінка зі `slug = ''`. Саме це робить перехід one-page → multi-page безболісним: не міняємо модель, а додаємо рядки в `pages`.

Орієнтовна форма (закріплено як еталон, деталі типів — на етапі дизайну БД):
```
tenants(id, host, canonical_hostname, nav_mode 'onepage'|'multipage', brand/футер..., ...)
pages(id, tenant_id, slug, page_type, title, is_published, show_in_nav, nav_order,
      UNIQUE(tenant_id, slug))            -- '' = home; захист від колізій slug
blocks(id, page_id, block_type, position, anchor, show_in_nav, nav_label, data jsonb,
       UNIQUE(page_id, anchor))
```
- `slug` зберігати **без** провідного/кінцевого слеша (`''` для home, `services/bouquets` для вкладеної).
- **ІНВАРІАНТ:** зарезервовані slug — єдиними константами. Зверни увагу на розподіл: matcher middleware виключає `sitemap.xml`/`robots.txt` та статику **за розширенням** (крапка в імені), а **іменний** список (`api`, `_next`, `admin`…) звіряється у **валідації конструктора + БД-unique** (у matcher вони присутні лише імпліцитно, не дослівно).

> *Відкинуто:* окрема модель під one-page без сутності «сторінка» — заблокувала б декларований перехід до multi-page. Журнал #4.

### 5.2. ✅ Дворівневий резолвинг
```
Host  → який це tenant      (kvity.com.ua → клієнт "Квіти")  [middleware → canonicalHostname → rewrite /_sites/{host}/...]
path  → яка сторінка tenant (/ → головна, /contacts → контакти)  [optional catch-all [[...slug]]; slug?.join('/') ?? '']
```
- `[[...slug]]` (optional catch-all) матчить і корінь (`slug: undefined`), і будь-яку вкладеність. **ІНВАРІАНТ:** `params` у App Router (Next 15+) асинхронні — спершу `const { host, slug } = await params`, не звертатися до `params.slug` синхронно. Неіснуючий slug → **явний `notFound()`** після запиту в БД (не через `dynamicParams`).
- `export const dynamicParams = true; export async function generateStaticParams() { return [] }` — порожній масив **обов'язковий** (інакше роут стає повністю динамічним, без ISR). Тенанти/сторінки рендеряться on-demand при першому візиті й кешуються.
- ⚠️ **matcher middleware** мусить виключати `/api`, `/_next`, файли з розширенням (`sitemap.xml`, `robots.txt`, статика): `matcher: ['/((?!api|_next|[\\w-]+\\.\\w+).*)']`. Інакше middleware перепише `/sitemap.xml` і зламає його.
- ⚠️ Catch-all ловить будь-який сміттєвий глибокий URL → кожен робить запит у БД + `notFound()`. Закласти захист (max-глибина / rate-limit), щоб не довбали БД.

### 5.3. ✅ Навігація — похідна, її НЕ редагують руками
Прямий наслідок рішення «поля, не канвас», і природний наслідок «сторінка = масив блоків» (4.1). Власник не має «редактора меню», який можна зламати — нав генерується з наявних даних, перемикач `tenants.nav_mode`:
- **односторінковий сайт** → пункти нав = **видимі секції** (блоки home з `anchor` + `show_in_nav`, anchor `#services`, `#about`). Сховав секцію (3, hide/show) → пункт зникає з меню автоматично.
- **багатосторінковий** → пункти нав = **опубліковані сторінки** (`pages` з `show_in_nav`, сорт за `nav_order`). Додав/прибрав сторінку → меню оновилось саме.
- Меню завжди консистентне, бо це проєкція даних, а не окрема редагована сутність.

### 5.4. 💡 One-page за замовчуванням (MVP)
**Рекомендація:** дефолт — **односторінковий** сайт (секції + anchor-нав у хедері), не мультисторінковий.
- Для локального бізнесу 50+ одна сильна сторінка (hero / послуги / галерея / контакти) закриває потребу і добре лягає під локальне SEO.
- Нав = скрол до секції, завжди працює; нема биття посилань і осиротілих сторінок; простіша модель для власника; вища надійність генерації.
- **Мультисторінковість** (каталог, окремі сторінки послуг, блог) — закладена в модель даних як **можливість** (5.1.1), але **не в MVP**. Структура `tenant → набір сторінок → набір блоків` уже не унеможливлює перехід пізніше.

## 6. Домени

Поняття «домен» розпадається на **три різні за складністю задачі**. Не плутати їх.

### Задача 1 — Піддомен (`name.vitryna.com.ua`)
✅ **ВИРІШЕНО: дефолт для всіх.** Тривіально: ми володіємо батьківським доменом, ставимо wildcard `*.vitryna.com.ua` на платформу, роутимо за `Host`, SSL покриває wildcard-сертифікат. Жодного зовнішнього API. Покриває ~80% аудиторії, видається безкоштовно й миттєво.

### Задача 2 — Підключити власний домен клієнта (який у нього вже є деінде)
✅ **Рішення: Cloudflare for SaaS / Custom Hostnames.** Клієнт додає CNAME → один POST на `/zones/{zone_id}/custom_hostnames` → Cloudflare сам випускає й поновлює SSL. Безкоштовний тариф: 100 custom hostnames.
- **Статус у продукті:** запасний шлях «у мене вже є домен». НЕ основний флоу (див. нижче).
- ⚠️ **Економіка на масштабі:** «100 безкоштовно» — не відповідь на 5000 клієнтів. Понад ліміт — ≈$0.10/host/міс, тож навіть ~1000 власних доменів (≈900 платних) ≈ $90/міс — це радше шум, ніж суттєва стаття; але рядок існує й множиться на к-сть клієнтів із власним доменом — врахувати в юніт-економіці (11). Ціну звірити перед фіксацією (тарифи Cloudflare змінюються).

### Задача 3 — Зареєструвати НОВИЙ домен за клієнта
✅ **Зона: `.com.ua` (третій рівень).** Без торгової марки, швидко, дешево.
⛔ **НЕ `.ua` (другий рівень):** вимагає сертифікат торгової марки + напівавтоматична модерація адміністратором 1–14 днів. Для нашої аудиторії це смерть конверсії — автоматизувати неможливо.
- Реалізація: через **reseller-API українського реєстратора** з передоплаченим балансом. Міжнародні реєстратори (Namecheap/GoDaddy) `.com.ua` через API зазвичай не вміють.

**Порядок розробки доменів:** Задача 1 → Задача 2 → (лише за платним попитом) Задача 3.

## 7. ✅ Реселінг-флоу реєстрації домену

**Контекст рішення:** клієнту байдуже, *де* зареєстрований домен — йому треба, щоб домен був **підключений до сайту**. Це спрощує архітектуру: якщо домен реєструємо ми, то **ми володіємо nameservers**, і вся CNAME-верифікація з Задачі 2 відпадає для основного флоу.

**Флоу (для клієнта — одна кнопка):**
1. **Чек доступності** — бекенд бʼє reseller-API, повертає вільно/зайнято + ціну.
2. **Альтернативи, якщо зайнято** — ⚠️ критично для аудиторії: введуть `kvity`, буде зайнято. Одразу пропонувати `kvity-lviv`, `kvitylviv`, `kvity2026`. Без цього людина застрягає.
3. **Реєстрація одним викликом** — домен + контакти реєстранта + **наші nameservers** (Cloudflare). Списується з передоплаченого балансу.
4. **DNS вже наш** — додаємо A/CNAME на платформу через Cloudflare API, SSL автоматом. Клієнт нічого не торкається.
5. **Запис у БД** — домен, реєстрант, дата закінчення, ID домену в реселера → у чергу поновлення. (Підключення власного/нового домену → оновити `canonicalHostname` tenant і зробити 301 з піддомену, див. 10.2.)

🔶 **ВІДКРИТЕ — головне рішення перед кодом: хто юридичний реєстрант домену.**
«Без різниці, *де* зареєстрований» (операційно) і «*на кого* зареєстрований» (WHOIS-власник) — різні речі. Для клієнта обидва варіанти виглядають однаково (одна кнопка); різниця проявляється через 1–2 роки.

| Варіант | Плюс | Мінус |
|---|---|---|
| Реєстрант = наш реселер-акаунт | Максимально просто | Клієнт у заручниках; єдина точка відмови (проблеми з акаунтом = всі домени разом) |
| Реєстрант = клієнт (через той самий API) | Домен юридично його; чистий вихід через auth-code; жодної міни | Трохи складніше технічно |

💡 **РЕКОМЕНДАЦІЯ (не підтверджена):** операційно — повний автомат (клієнт нічого не бачить); юридично — **реєстрант = клієнт**. Для `.com.ua` вимоги легкі (контактні дані, без ТМ), тож це реально. Виняток — якщо свідомо обираємо lock-in як стратегію утримання.
⚠️ Переграти постфактум дорого: 5000 доменів на нашому акаунті → «віддати клієнтам» = 5000 трансферів.

🔶 **ВІДКРИТЕ:** хто платить за поновлення і що відбувається при відтоку. Потрібне автосписання в гривні + нагадування, інакше через рік сайти масово відвалюються і бʼють по нас. (Зчеплено з білінгом, 11: щоб поновлювати від імені клієнта, однаково треба токенізована картка + dunning + грейс-період — проєктувати як одну підсистему.)

## 9. ✅ Хостинг і модель рендеру

**Вартість визначає НЕ провайдер, а модель рендеру.**

Сайти аудиторії — низькотрафічні, майже не змінюються (телефон раз на місяць). Тому:

✅ **Рендер-модель:** рендеримо сайт у **статику/кеш**, віддаємо з CDN, інвалідуємо кеш конкретного клієнта **тільки на «Зберегти»** (ISR / on-demand revalidation / purge-on-save). Вартість на візит → майже нуль.
⛔ **НЕ SSR на кожен запит** — інакше платимо compute за кожен перегляд кожного з тисяч сайтів.

### 9.1. ✅ Інвалідація — per-tenant через теги `🔶→✅ (2026-06-30)`
Уточнення «purge-on-save» за підсумком валідації:
- **ІНВАРІАНТ:** інвалідація **per-tenant через теги** (`tenant:{host}`, за потреби `page:{host}:{slug}`), а **не** через шлях-патерн. `revalidatePath('/_sites/[host]/...', 'page')` з динамічним сегментом `[host]` інвалідує кеш **усіх** тенантів — пастка. `revalidatePath` лишити лише для **літерального** destination-шляху конкретної відкритої сторінки.
- Ключ ISR-кешу = **переписаний destination-шлях** (`/_sites/{host}/{slug}`), тож різні сторінки одного тенанта кешуються окремо автоматично.

Базовий (стабільний Next 15) патерн:
```ts
const getPage = (host, slug) => unstable_cache(
  () => db.pages.find(host, slug),
  ['page', host, slug],
  { tags: [`tenant:${host}`, `page:${host}:${slug}`] }
)()
// на «Зберегти»:
revalidateTag(`tenant:${host}`)        // весь тенант, не зачіпаючи інших
revalidateTag(`page:${host}:${slug}`)  // одна сторінка
```

⚠️ **Звірити перед інтеграцією — залежить від версії Next.** Наведений патерн орієнтований на **стабільний Next 15.x**, де `unstable_cache` + теги + **одно-аргументний** `revalidateTag` — стабільний шлях, а Cache Components (`'use cache'` / `cacheTag` / `cacheLife`, **дво-аргументний** `revalidateTag(tag, profile)`, `updateTag`) — canary. **У Next 16+ ситуація інвертувалася:** одно-аргументний `revalidateTag(tag)` стає **deprecated**, базовий підпис зміщується на дво-аргументний `revalidateTag(tag, 'max')` + `updateTag` (`revalidateTag(tag, {expire:0})` для негайної інвалідації). **Зафіксувати цільову версію Next перед кодом** і звірити cache-API по docs саме під неї — не кодити з пам'яті/туторіалів.

### 9.2. Стек-провайдер
🔶 **ВІДКРИТЕ** (вердикт-схильність є, не зафіксовано):

| Варіант | За | Проти |
|---|---|---|
| **Cloudflare** (Workers + Cloudflare for SaaS) | Найвирівняніший по вартості; домени вже тут; custom hostnames рідні (100 безкоштовно); дешевий egress; tenant-резолвінг у Worker по `Host` | ⚠️ Next.js на Workers (OpenNext) менш зрілий: ISR/on-demand revalidation/middleware/кеш мають гострі кути |
| **Vercel** (Vercel for Platforms) | Створений під цей сценарій; найкращий DX під Next.js; custom domains через Domains API; tenant-резолвінг у middleware; ISR рідний | Вища ціна на масштабі |

💡 **Вердикт:** при гривневій підписці (економимо копійки) → **Cloudflare-сторона** по вартості. АЛЕ ⚠️ міграція мультитенантного роутингу + ISR між платформами **дорога й ризикована** — «почнемо на CF, мігруємо потім» наївне. Прагматичний гібрид: **застосунок на Vercel (Vercel for Platforms), а домени/SSL/custom hostnames/R2 — на Cloudflare**. Не плати DX-податком у рендер-шарі там, де він найдорожчий. Актуальні ліміти/ціни звірити перед фіксацією.

**Не плутати з «хостингом застосунку»:**
- ✅ **Дані:** Postgres у **Supabase**, tenant-ізоляція через **RLS**. *(Уточнення: RLS — площина редагування/дашборду. Публічний рендер майже завжди потрапляє в кеш, а не в БД, і читає published-контент сервісним ключем — не плутати RLS з ізоляцією публічного рендеру.)*
- ✅ **Зображення:** **Cloudflare R2** (нульовий egress — критично, бо сайти image-heavy і тисячі сайтів роздають фото). Supabase Storage робочий, але R2 виграє на egress.

**Повний стек end-to-end:**
```
один Next.js застосунок (Vercel; домени/SSL/R2 на Cloudflare)
  → middleware: Host → canonicalHostname → rewrite /_sites/{host}/...
  → tenant config (хедер/футер/нав/тема-токени) + блоки сторінки за slug
  → контент із Supabase (Postgres + RLS), читання обгорнуте в кеш із тегами tenant/page
  → зображення з Cloudflare R2
  → рендер у кеш (ISR on-demand), purge-on-save через revalidateTag(`tenant:{host}`)
  → домени й SSL на Cloudflare
```

## 10. ✅ SEO per-tenant

SEO — це **проєкція тих самих даних**, per-tenant, а не окрема фіча, що вмикається збоку. Той самий принцип, що з навігацією: похідна від полів, які клієнт уже заповнив.

**Артефакти, що генеруються на кожен клієнтський сайт (з його ж даних):**
- `<title>` + `meta description` — з назви бізнесу, міста, категорії, послуг. Через `generateMetadata` (Server Component), host з **`params`** (2.4), `metadataBase` **per-tenant** з `canonicalHostname`, URL-поля Metadata API **відносні** (Next зробить абсолют на правильний хост) — **крім JSON-LD: там абсолюти задавати вручну з `canonicalHostname`** (не проходить через `metadataBase`, див. 10.1).
- Open Graph / Twitter card — щоб лінк гарно розгортався у Viber/Telegram/Instagram (канали аудиторії). Динамічні OG-зображення per-tenant через `opengraph-image.tsx` + `ImageResponse` (`next/og`), host теж із **`params`** (інакше зображення стане динамічним і дорогим).
- `sitemap.xml` і `robots.txt` — окремі **на кожен hostname** (`app/sitemap.ts` / `app/robots.ts`, host із `headers()` — єдине легітимне місце; URL будувати з `canonicalHostname`).
- **JSON-LD `LocalBusiness`** (Florist / HairSalon / LegalService…) — назва, адреса, гео, години, телефон, ціновий діапазон. Обов'язкові: `name`, `address`(PostalAddress). Для локального бізнесу — головний шматок.
- `canonical` — на `canonicalHostname`.

### 10.1. ⚠️ Пастка 1 — host-absolute URLs
Для Google кожен hostname — окремий сайт, хоча застосунок один. Canonical, посилання в sitemap, `og:url`, поле `url`/`image` у structured data будувати з **`canonicalHostname`**, **не** з глобального домену і **не** з host запиту. Інакше отруєна індексація клієнта.
- **ІНВАРІАНТ:** глобальний `metadataBase` на кореневий домен **заборонено** — він канонікалізує всі сайти на один хост. Будувати `metadataBase` per-tenant у `generateMetadata`.
- ⚠️ JSON-LD `url`/`image` **не проходять** через `metadataBase` — задавати абсолюти вручну з `canonicalHostname`, стежити, щоб збігалися з canonical.

### 10.2. ⚠️ Пастка 2 — дублі по двох hostname
Якщо сайт доступний і на піддомені, і на власному домені — це дубльований контент.
- ✅ **Рішення `🔶→✅ (2026-06-30)`: 301 з піддомену на `canonicalHostname` у middleware** (директива), а не лише canonical-тег (підказка, яку Google може зігнорувати при ідентичному контенті). `canonicalHostname` — поле в моделі tenant (5.1). Коли клієнт підключає/реєструє власний домен → оновити `canonicalHostname` + увімкнути 301.

> *Відкинуто:* покладатися лише на `rel=canonical` для дублів — слабше за 301; не консолідує вагу надійно. Журнал #9.

### 10.3. Локальне, а не класичне SEO
Львівська квіткарка не виграє по запиту «квіти» і їй це не треба. Працює:
- **Google Business Profile** (map pack) — головний канал, де локальний бізнес реально знаходять.
- **NAP-консистентність** — назва/адреса/телефон ідентичні на сайті й у GBP.
- місто + категорія в `title`/`H1` («Флористика у Львові»).
- Роль сайту: канонічна веб-присутність, яку GBP підтверджує. Для цього сегмента інтеграція з GBP важливіша за класичну on-page.

**Синергія (вже безкоштовно):** статика/кеш (9) → швидке завантаження → добрі Core Web Vitals → ранжуючий сигнал. Архітектура вже працює на SEO.

⚠️ **Чесне обмеження (не продавати магію):** свіжий домен + тонкий AI-текст не вибʼються в топ швидко — у нового домену нуль авторитету, органіка росте місяцями. Реальний швидкий виграш — **GBP-лістинг + технічно чистий індексований сайт**, а не «топ по запиту». Очікування власника керувати в бік локального/GBP.

### 10.4. ⚠️ Ризик масштабу — scaled-content-abuse / репутація кореня `🔶→✅ (2026-06-30)`
Тисячі структурно схожих сайтів із тонким AI-текстом на піддоменах **одного кореня** `vitryna.com.ua` підпадають під Google **scaled content abuse** (заборонено не «AI», а «багато сторінок без цінності для користувача») і **ділять репутацію кореневого домену**. Мітигація (закладати з MVP, не пост-фактум):
- Більше **структурованих фактів** (години/адреса/ціни/фото/відгуки), менше «AI-води»; описи з реальних даних, а не один темплейт із підстановкою назви.
- **`noindex` для порожніх/демо/чернеткових тенантів** — головний запобіжник проти маси тонких сторінок.
- **Підштовхувати активних клієнтів на власні домени + 301 з піддомену** → власний домен **ізолює** репутацію від кореня. Додатковий аргумент за апсел власного домену (6–7).
- Не генерувати сторінки «послуга X у місті Y» сотнями без реального бізнесу за ними.

**Звʼязок із доменом:** власний домен дає чистішу індексацію, ізоляцію репутації й брендову присутність → це підсилює апсел власного домену (6–7). Спільний піддомен — слабше й менш солідно.

🔶 **ВІДКРИТЕ:** інтеграція з Google Business Profile не спроєктована.

## 11. ⚠️ Наскрізні ризики (усвідомлені, рішень нема)

- 🔶 **Білінг:** рекурентні платежі в гривні складніші за Stripe (токенізація карт, автосписання, dunning/повторні спроби при відмові). Кандидати: LiqPay / WayForPay / Fondy. Провайдер не обрано. **Зчеплено з поновленням домену (7)** — проєктувати як одну підсистему зі станами `активний → прострочка → грейс → призупинено → видалено`. Елевація: це не «ризик на потім», а кандидат на **прототип №1**, бо гейтить усю модель доходу.
- ⚠️ **Підтримка:** нетехнічна аудиторія = високе навантаження на підтримку («чому не працює», «як поміняти фото»). Операційка вбиває такі продукти частіше за технологію. Часткова мітигація вже в архітектурі: field-модель (3) звужує поверхню «що можна зламати».
- ⚠️ **Юніт-економіка:** домен + хостинг + AI-генерація (особливо картинки) + еквайринг + custom hostnames понад безкоштовний ліміт (6, Задача 2) + (можливі сповіщення) проти невисокої гривневої підписки = тонка маржа. Модель не порахована.

> SEO як ризик «власник очікує бути в Google» закрито окремим розділом 10 (відповідь: керувати очікуванням у бік локального/GBP). Ризик scaled-content-abuse — у 10.4.

## 13. Рекомендовані наступні кроки

> Переприоритезовано за підсумком валідації: спершу де-ризикувати core-loop цінності, потім домени (це апсел, не основна петля).

1. **End-to-end на ОДНІЙ вертикалі (квіткарня):** input → класифікація вертикалі → вибір пресету → генерація (реєстр блоків + grounding) → редагування полів → **перемкнути шаблон** → завантаження/імпорт фото. Доводить, що петля цінності працює. (Розділи 4, 3, 5.)
2. **Прототип рекурентного UAH-білінгу** (WayForPay/LiqPay/Fondy) з токенізацією + dunning; спроєктувати спільний state machine білінг+домен (11, 7).
3. **Онбординг із префілом** з Facebook/Instagram/GBP (зменшує введення для 50+ і частково закриває проблему фото).
4. Тільки **субдомен** (Задача 1) у MVP; власний домен/реєстрацію (Задачі 2–3, реєстрант, реселер) відкласти до платних юзерів.
5. Спроєктувати модель даних: `tenant config` (бренд/нав/футер/`canonicalHostname`/тема-токени) × `сторінки` × `блоки` (розділи 5, 4.1, 7).
6. Спроєктувати першу field-схему під квіткарню як еталон + її SEO-проєкцію (розділ 10) + реєстр блоків під неї (4.1).
7. *(після core-loop)* Зафіксувати провайдера хостингу (гібрид Vercel + Cloudflare, 9.2) + цільову версію Next (9.1).
8. *(при платному попиті на домени)* Вирішити реєстранта домену (7) — блокує доменну підсистему, але не критичний шлях MVP.
9. Описати request-flow end-to-end (Host → canonicalHostname → page → render → cache → invalidate on save).
